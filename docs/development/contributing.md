# Контрибьюшн гайдлайн

## Цель
Определить правила внесения изменений, чтобы кодовая база оставалась стабильной и читаемой.

## Основные правила
- Создавайте отдельные ветки под задачи.
- Следуйте соглашениям проекта (ESLint/Prettier, Tailwind, именование файлов).
- Добавляйте документацию и обновляйте `docs/` при изменениях в архитектуре или API.

## Процесс
1. Подготовьте изменения и убедитесь, что сборка проходит локально.
2. Обновите документацию/чек-листы при необходимости.
3. Оформите понятное сообщение коммита и описание PR.

## Связанные модули
- `docs/` — актуальная документация.
- `src/app/` — основной код приложения.
- `prisma/` — изменения схемы БД.

## Исторические материалы
Отчёты в `docs/reports/` могут содержать устаревшие решения и служат для контекста развития проекта.
